# GodaPlayer â€” Android Audio Player PRD

## Overview

GodaPlayer is a native Android application for playing and organizing local audio files. The app provides robust playlist management capabilities with a retro terminal aesthetic. All functionality operates offline with local storage only â€” no network, cloud, or streaming features.

### Core Value Proposition

| Feature | Description |
|---------|-------------|
| **File Browser** | Navigate device storage to find and select audio files |
| **Playback Engine** | Full-featured audio playback with standard controls |
| **Playlist Management** | Create, edit, organize, and manage multiple playlists |
| **Equalizer** | Built-in audio equalizer with presets and custom bands |
| **Offline-First** | 100% local operation, no internet required |

### Supported Audio Formats

| Format | Extension | Notes |
|--------|-----------|-------|
| MP3 | .mp3 | MPEG Audio Layer III |
| FLAC | .flac | Lossless compression |
| AAC | .aac, .m4a | Advanced Audio Coding |
| OGG | .ogg | Ogg Vorbis |
| WAV | .wav | Uncompressed audio |

### Visual Aesthetic

GodaPlayer uses a retro terminal aesthetic throughout:

- **Dark theme** as the primary (and only) color scheme
- **Monospace typography** for all text elements
- **High-contrast** text on dark backgrounds
- **ASCII-style decorations** and box-drawing characters where appropriate
- **Minimal color palette** â€” accent color for interactive elements, muted tones elsewhere

## Performance Considerations

### File Scanning

- **Background scanning:** Use WorkManager for periodic library scans
- **Incremental updates:** Compare file modification dates, skip unchanged files
- **ID3 parsing:** Use MediaMetadataRetriever, cache results in database
- **Progress feedback:** Show scan progress for large libraries

### Memory Management

- **Lazy loading:** Load song metadata on-demand for large lists
- **Thumbnail caching:** Cache extracted album art with size limits
- **Paging:** Use Paging 3 library for lists exceeding 500 items

### Battery Optimization

- **Foreground service:** Required for background playback
- **Wake locks:** Minimal use, release promptly
- **Efficient scanning:** Respect battery saver mode

## MVP Scope

### Phase 1: Core Playback (Week 1-2)

- [ ] Project setup, architecture scaffolding
- [ ] Basic file browser with folder navigation
- [ ] Support for MP3, FLAC, AAC, OGG, WAV formats
- [ ] Audio playback with ExoPlayer
- [ ] Now Playing screen with basic controls
- [ ] Play/pause, next, previous
- [ ] Background playback with notification
- [ ] Basic queue management

### Phase 2: Library & Database (Week 3-4)

- [ ] Room database setup
- [ ] File scanning with metadata extraction (ID3, Vorbis comments, etc.)
- [ ] All Songs view with sorting
- [ ] Search functionality
- [ ] Play count and last played tracking
- [ ] Duplicate handling (same file = one entry; same name/artist but different file = separate entries)

### Phase 3: Playlist Management (Week 5-6)

- [ ] Create/rename/delete playlists
- [ ] Add songs to playlists (single and batch)
- [ ] View playlist contents
- [ ] Reorder tracks within playlist
- [ ] View which playlists contain a song
- [ ] Remove songs from playlists
- [ ] Duplicate playlist
- [ ] Large playlist warning (performance notice for 500+ songs)

### Phase 4: Equalizer (Week 7)

- [ ] 5-band equalizer implementation
- [ ] Bass boost control
- [ ] Virtualizer control
- [ ] 12 built-in presets
- [ ] Custom preset save/load/delete
- [ ] EQ on/off toggle
- [ ] Persist settings across sessions

### Phase 5: Polish & Settings (Week 8-9)

- [ ] Shuffle and repeat modes
- [ ] Gapless playback
- [ ] Settings screen
- [ ] M3U export/import
- [ ] Error handling and edge cases
- [ ] Performance optimization
- [ ] UI polish and animations

## Design Decisions (Resolved)

1. **Album art handling** â€” Deferred to post-MVP. For MVP, use a default visualization/icon for all tracks.

2. **Duplicate handling** â€” Files are identified by their full file path. If the exact same file exists in multiple locations, treat as separate entries. If two files have the same artist and title metadata but are different files (e.g., live version vs. studio version), they are naturally separate entries since they have different file paths. This keeps the implementation simple while preserving user intent.
